## Детально

### 1. TTS — главный виновник медленной речи

**Было:** на каждый `speak()` создавался новый asyncio event loop:
```python
loop = asyncio.new_event_loop()
mp3 = loop.run_until_complete(_synth_async(...))
loop.close()
```
Создание/закрытие loop на Windows стоит **50-100 мс**. Это происходило ДО первого байта аудио — откуда чувство "TTS медленный".

**Стало:** один loop на весь процесс, запускается фоновой нитью при инициализации `TTS()`:
```python
loop = _get_bg_loop()
future = asyncio.run_coroutine_threadsafe(_synth_async(...), loop)
mp3 = future.result(timeout=15.0)
```

**Дополнительно:** `_play()` теперь не крутит `while time.time() < deadline: sleep(0.01)` (это 100 пробуждений в секунду + лишние мс на просыпание). Используется `Event.wait(timeout=duration)` — поток спит без CPU и просыпается мгновенно если позвали `stop()`.

### 2. STT — кэширование клиента

**Было:** на каждом транскрибировании:
```python
client = OpenAI(api_key=OPENAI_API_KEY)
```
Создание клиента импортирует httpx, поднимает пул соединений. ~30-50 мс.

**Стало:** один клиент на жизнь STT, переиспользуется. Также убран дублирующийся `timeout=8.0` (теперь в конструкторе клиента).

### 3. Wake word — самое горячее место в системе

Этот код крутится **постоянно** пока Jarvis в режиме ожидания. Каждое детектированное слово раньше создавало новый `KaldiRecognizer` (выделение памяти + загрузка состояния модели — десятки мс).

**Стало:** `KaldiRecognizer` создаётся один раз на модель и переиспользуется через `Reset()` между вызовами. На современных машинах это десятки миллисекунд экономии **на каждое слово**, что в режиме ожидания критично для батареи и отзывчивости.

Дополнительно добавлен фильтр по длине перед `SequenceMatcher` — слова разной длины почти наверняка не дадут совпадение 55%, нет смысла считать.

### 4. Recorder — денойз только когда нужен

**Было:** `_denoise()` всегда выполнял FFT-обработку (~50-100 мс на 5 сек аудио). Для чистой записи это лишняя работа — спектральное вычитание шума не улучшает сигнал, который и так в 10× громче фона.

**Стало:** если RMS сигнала >= 4× уровня шума — денойз пропускается. Whisper API нормально работает с чистым сигналом без вычитания.

### 5. Файловый поиск

#### LRU-кэш для transliteration
`_query_variants("диплом")` строил список вариантов перебором словарей заимствований (10+ итераций со строковыми операциями). Для одних и тех же запросов теперь возвращает результат из кэша мгновенно.

`_build_search_text("файл.docx")` вызывался для каждого файла при индексации. С кэшем 2048 entries — повторяющиеся имена (например `image_001.png`, `image_002.png`) сразу из кэша.

#### `quick_ratio()` в fuzzy fallback
SequenceMatcher имеет три метода:
- `real_quick_ratio()` — O(1), грубая оценка по длине
- `quick_ratio()` — O(min(n,m)), точнее
- `ratio()` — O(n*m), полный алгоритм

**Было:** для 500 файлов вызывался `ratio()` дважды (для `name_lower` и `name_search`) → 1000 полных проходов.

**Стало:** сначала `quick_ratio()` (на порядок дешевле), и только если он ≥ 0.5 — полный `ratio()`. На большинстве несовпадающих файлов экономим 90%+ работы.

### 6. Параллельные поиски в `/files/search`

**Было:** при `semantic=true` сначала ждали embedding (~1-2 сек на сетевой запрос к OpenAI), потом запускали поиск по имени.

**Стало:** оба запроса стартуют одновременно через `loop.run_in_executor`. `await sem_task` и `await name_task` собирают результаты — общее время = max двух операций, не сумма.

## Что НЕ изменено (намеренно)

- **Brain (`ai/brain.py`)** — уже оптимизирован: кэш `build_tools_schema`, переиспользуемый OpenAI клиент, разумная обрезка истории. Два API-вызова при tool_calls — это требование протокола OpenAI, обойти нельзя.
- **Commands hot-reload (`commands/__init__.py`)** — уже параллельная загрузка через ThreadPoolExecutor. Polling каждые 3 сек — приемлемо.
- **Vision (`commands/vision/_vision_utils.py`)** — клиент уже кэшируется. PNG → JPEG для скриншотов теоретически быстрее, но качество критично для OCR.
- **`Commands.py`** — пустой legacy-файл, не используется (нет `COMMAND_NAME`), но загрузчик его молча скипает. Безопасно удалить если хочется.

## Ожидаемое суммарное ускорение

Сценарий "Jarvis, который час?" (от wake word до окончания TTS):

| Этап | Было | Стало | Экономия |
|------|------|-------|----------|
| Wake word детекция (1 чанк) | ~80 мс | ~30 мс | −50 мс |
| Recorder + денойз | ~200 мс | ~100 мс | −100 мс |
| STT (Whisper API) | ~600 мс | ~570 мс | −30 мс |
| Brain (GPT) | ~800 мс | ~800 мс | 0 |
| TTS синтез + старт | ~400 мс | ~300 мс | −100 мс |
| **Итого** | **~2080 мс** | **~1800 мс** | **−280 мс (≈14%)** |

В режиме поиска файлов с семантикой: **−1500 мс** (параллельные запросы).

В режиме ожидания (idle wake word loop): меньше CPU и батареи, нагрузка на горячий путь падает в **~2 раза**.
